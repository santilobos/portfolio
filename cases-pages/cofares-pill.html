Building a Framework Agnostic Design System by Cofares

An agnostic design system is an ideal that allows us to design and implement once with the focus always on multiple platforms. This project was born with the objective of helping product teams, both designers and developers, to speak the same language favoring consistency and scalability of products while reducing effort and time, which translates directly into substantial cost savings.

THE CHALLENGE

COFARES is the leading company in the distribution of medicines in Spain. A cooperative for the distribution of medicines, health products and services with pharmaceutical capital. This distribution and sale of medicines is done through different digital channels, and it is right here where the main problem lies. Each product is designed and developed by different teams using different methodologies and technologies.

For this reason I was hired to build from scratch the new design system of the company, a system that is able to provide consistency and scalability to all products and teams regardless of the technology used. This is where the new technology agnostic design system was born.

Overview

In an environment of chaos and lack of structure and working procedures, the main task was to establish design principles and best practices across all products to ensure consistency and scalability of all products.

Problem definition

The lack of an aligned and systematized design and development methodology and the production of isolated screens with a short-term vision generate problems when it comes to validating the user experience and product iterations.

Methodology

We work following the atomic design methodology created by Brad Frost, which allows us to build consistent and scalable design systems, aligning the different work teams and considerably saving effort and budget.

Design proposal

We create a library of components based on foundations and design tokens communicated through the figma api with the development repository where they will be built using html, css and javascript.

Building our design system in a structured and efficient way through Atomic Design

Atomic Design is a methodology for creating modular and scalable design systems, proposed by Brad Frost. It is based on the idea of building user interfaces by breaking them down into reusable components, similar to how atoms form molecules and organisms in nature. Atomic Design allows the development of consistent design systems, where each piece is clearly defined and can be reused in different contexts.

They apply to component creation, governance, maintenance, and being a teammate that supports design systems. Defining the bases of our design system

Foundations are the styles, effects and structure that govern the creation of components within the design system. They are aligned with branding, accessibility and UX best practices to achieve an equitable and unified experience.

We work closely with the client to define these foundations according to the brand guidelines of each of the company's digital products. This is the result of the workshop.

Using BEM methodology for the definition of our design tokens

BEM (Block, Element, Modifier) is a naming methodology used to organize and structure CSS code in a clear and predictable way, and can be applied in the definition of design tokens. In the context of design tokens, BEM helps to name and categorize styles in a hierarchical manner, facilitating their identification and use in reusable components. Its main benefits include improved code readability, consistency in style mapping and scalability, which allows working on large projects without CSS becoming chaotic. It also promotes modularity and maintainability by keeping styles isolated and easily modifiable.

We create design tokens so they can be used on multiple platforms.

From the beginning it was clear to us that we wanted to define a workflow that would allow us to be agile when dealing with interface iterations in the products. To do this we synchronized the design tokens with the development variables through the Figma API, which allows us to make changes in one place to be modified in the other. This way we always keep the libraries up to date.

We also used the Style Dictionary tool, which allowed us to export the design tokens in variables of different languages to be used as needed.

One Set of Design Tokens to Rule Them All with Style Dictionary

Style Dictionary is a tool that allows you to manage and export design tokens to different platforms and languages, such as CSS, JSON, Android, iOS or web. Its main advantage is that it centralizes the definition of tokens, facilitating consistency in designs regardless of the technology used. In addition, it allows automating the export process, ensuring that changes are reflected in all platforms simultaneously. This improves scalability and design maintenance, as any updates to the tokens are applied in a unified manner. Finally, it ensures that design and development teams work with the same sources of truth.

Component Design Process: Ingredients to Achieve Functional Structures

The process of creating components in Figma begins with extensive research. First, we turn to the Mozilla Developer documentation to understand the structure and functionality of the components we wish to design. This documentation provides us with valuable information on best practices and essential elements to include.

Then, in Figma, we elaborate the properties and variants of the component, allowing us to define different states and configurations in a clear and organized way. Finally, we integrate all this information to create a user interface (UI) version of the component, ensuring that it is consistent, functional and ready for implementation in the final product. This methodical approach not only improves design quality, but also facilitates collaboration and long-term maintenance.

Design Tokens applied to Button Components: Consistency, Maintainability and Scalability

Using design tokens in the button components provides several key advantages. In the first component, which uses the pf namespace, and in the second, which uses wn, the tokens enable visual consistency by applying consistent styles across both buttons, even if they belong to different themes or layout variants. This simplifies maintenance, as any adjustments to the tokens are automatically reflected in both components, ensuring that all design changes are quick and effective. In addition, by using design tokens, collaboration between designers and developers is improved, as they can speak the same language in terms of styles.

Tokens also facilitate scalability, allowing the creation of new buttons or components in the future without the need to redefine styles from scratch. Finally, incorporating accessibility principles into tokens ensures that both buttons are functional and inclusive for all users.
